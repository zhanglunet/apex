# APEX V1.2 产品设计：Evidence Layer

## 1. 产品定位

APEX V1.2 的主题是 Evidence Layer。它不是新开一条产品线，而是在现有 R1 Meeting Intelligence 上补齐“证据可追踪”能力。

V1.1 已经把会议材料变成结构化工作流：

```text
上传材料 -> 创建任务 -> 生成输出 -> 人工修订 -> Failure Card -> Eval Case -> Memory Object
```

V1.2 要让这条链路继续升级：

```text
结构化输出 -> 证据项 -> 证据缺口 -> Failure Card -> Eval Case -> 回归验证
```

核心产品判断：

```text
研究工作流真正可用，不取决于输出有多完整，而取决于关键结论能否被审计。
```

## 2. 用户与场景

### 2.1 用户角色

| 角色 | 目标 | 主要风险 |
| --- | --- | --- |
| 研究负责人 | 快速判断会议输出是否可用 | 把无证据结论带入决策 |
| 分析师 | 修订输出并补充证据 | 漏掉关键行动项或来源 |
| 质量运营者 | 维护 Failure / Eval 闭环 | 失败样本无法转成可验证标准 |
| 决策支持成员 | 复用 Memory 和会议结论 | 不知道记忆从何而来 |

### 2.2 高频场景

1. 会议后，分析师生成 R1 输出，并查看 Evidence Panel。
2. 研究负责人只看缺证据和弱证据项，决定是否允许导出。
3. 质量运营者在 Evidence Workbench 中筛选 `MISSING` 项，批量转成 Failure Card。
4. 团队把证据缺口沉淀成 Eval Case，后续验证 Prompt 或模型是否改善。

## 3. 用户体验原则

1. 证据状态要一眼可见  
   `SUPPORTED`、`WEAK`、`MISSING` 必须有明确标签，不依赖用户读长文本判断。

2. 不阻断主流程  
   V1.2 不要求用户先修完所有证据才能导出，但要清楚暴露风险。

3. 不做虚假精确  
   当前没有逐字定位能力时，不展示“第几行第几段”这种假精确引用。先展示 `sourceHint` 和证据文本。

4. 缺口能进入闭环  
   发现证据缺失后，用户下一步应能创建 Failure Card 或 Eval Case，而不是只看到红色提示。

## 5. 核心页面设计

### 5.1 Run Detail：Evidence Panel

位置：Run Detail 的 Quality Panel 和 Memory Objects 附近。

内容：

- Evidence 总数。
- `SUPPORTED` / `WEAK` / `MISSING` 数量。
- Evidence Item 列表。
- 每条显示：
  - section
  - status
  - claim
  - evidenceText
  - sourceHint
  - 创建 Failure Card 操作

空状态：

```text
生成后会从关键变化、行动项和质量警告中提取 Evidence Items。
```

### 5.2 Evidence Workbench：`/evidence`

用途：集中运营证据缺口。

筛选：

- status：全部 / SUPPORTED / WEAK / MISSING
- section：全部 / KEY_CHANGE / ACTION_ITEM / OPEN_QUESTION / QUALITY_WARNING

列表字段：

- status
- section
- claim
- evidenceText
- source Run
- createdAt

关键操作：

- 查看任务。
- 创建 Failure Card。
- 后续可增加“标记已支持”。

### 5.3 Dashboard

新增指标：

- Missing Evidence
- Weak Evidence
- Supported Evidence

点击指标进入 `/evidence` 对应筛选。

## 6. 数据设计

### 6.1 EvidenceItem

```text
EvidenceItem
  id
  routeRunId
  section
  claim
  evidenceText
  sourceHint
  status
  createdAt
```

### 6.2 状态定义

| 状态 | 含义 | 示例 |
| --- | --- | --- |
| `SUPPORTED` | 有明确证据文本 | “客户提到 Q3 预算将重新评估” |
| `WEAK` | 有证据但不足以支撑结论 | “待确认” / “会议中提及” |
| `MISSING` | 无证据或明确缺失 | “缺少原文支持” |

## 7. 生成与抽取规则

V1.2 不改变 LLM JSON schema 的主结构，先从现有字段抽取：

- `keyChanges[].change` -> claim
- `keyChanges[].evidence` -> evidenceText
- `actionItems[].task` -> claim
- `actionItems[].evidence` -> evidenceText
- `openQuestions[]` -> claim，status 默认 `MISSING`
- `qualityWarnings[]` -> claim，status 由关键词判断

## 8. 失败闭环设计

当 Evidence Item 是 `MISSING` 或 `WEAK` 时，可以创建 Failure Card：

默认字段：

| FailureCard 字段 | 默认值 |
| --- | --- |
| `failureType` | `MISSING_EVIDENCE` |
| `severity` | `P1` for MISSING, `P2` for WEAK |
| `description` | `证据不足：{claim}` |
| `originalOutput` | 当前 Run 输出 |
| `userRevision` | 空或当前编辑输出 |

生成 Eval Case 时，期望行为应包含：

```text
新输出必须为该 claim 提供证据；如果原始材料不支持该 claim，应将其降级为开放问题或写入 Quality Warnings。
```

## 9. 验收指标

### 9.1 产品验收

- 用户可以在 Run Detail 中看到证据项。
- 用户可以在 `/evidence` 中集中查看缺证据项。
- 用户可以从证据缺口进入 Failure/Eval 闭环。
- Dashboard 能显示证据健康状态。

### 9.2 工程验收

- `npm run build` 通过。
- `npm run test:smoke` 通过。
- 重复生成不会重复写入 EvidenceItem。
- `/api/health` 正常返回。

### 9.3 数据验收

- 每次生成后 EvidenceItem 数量可解释。
- `MISSING` 不应被误标为 `SUPPORTED`。
- `待确认`、`缺少证据`、空字符串应归为 `WEAK` 或 `MISSING`。
